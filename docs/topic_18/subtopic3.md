# Інтеграція бізнес-логіки та обробники подій

При створенні графічних додатків типовою помилкою багатьох початківців є змішування візуального інтерфейсу з основною бізнес-логікою програми (обчисленнями, роботою з базою даних, алгоритмами). Таке явище в розробці називають **"Code-behind problem"**, або розміщення всього у файлі форми.

## Що таке "бізнес-логіка"?

**Бізнес-логіка** (Domain Logic) — це частина коду, яка виконує специфічні задачі вашого застосунку, незалежно від того, яким чином ця програма взаємодіє з користувачем.

- Наприклад, якщо ви пишете програму-калькулятор, розрахунок відсотків або квадратних коренів — це бізнес-логіка.
- Зчитування тексту з елемента `TextBox` чи зміна кольору форми — це **логіка представлення** (UI Presentation Logic).

### Проблема "Товстої форми"

Якщо ви помістите код підключення до бази даних та складні обчислення прямо в обробник `btnSave_Click` у `Form1.cs`, ви отримаєте:

1. **Неможливість повторного використання.** Якщо ви захочете додати консольний інтерфейс або веб-сайт до свого застосунку, код доведеться писати чи копіювати заново.
2. **Складність тестування.** Ви не зможете написати модульні тести для цього коду, не ініціалізуючи графічну форму повністю.
3. **Плутанину у коді.** Код стане величезним, в ньому буде важко розібратися та супроводжувати його.

## Підключення обробників подій для інтеграції логіки

Правильний підхід до розробки передбачає винесення бізнес-логіки в окремі класи. Форма (`Form`) має бути лише тонким прошарком (View), мета якого:

1. Отримати дані від користувача з `TextBox`, `CheckBox` тощо.
2. Передати ці параметри у метод класу-сервісу з відокремленою бізнес-логікою.
3. Отримати від сервісу результат.
4. Відобразити отриманий результат у потрібних елементах (наприклад, у `Label` або `DataGridView`).

### Архітектурний приклад

Уявімо застосунок розрахунку знижок.
Спочатку ми створюємо окремий (незалежний від `Windows.Forms`) клас зі своєю логікою.

```csharp
// Окремий файл: DiscountCalculator.cs
public class DiscountCalculator
{
    public decimal CalculateDiscount(decimal totalAmount, bool isVipCustomer)
    {
        if (totalAmount < 0) throw new ArgumentException("Сума не може бути від'ємною");

        decimal discountRate = 0;

        if (totalAmount > 1000) discountRate = 0.05m;
        if (totalAmount > 5000) discountRate = 0.10m;

        if (isVipCustomer) discountRate += 0.05m;

        return totalAmount - (totalAmount * discountRate);
    }
}
```

Тепер, у самій формі, ми пишемо обробник події кліку для кнопки "Розрахувати", який виступає просто "посередником" між графічним інтерфейсом та калькулятором:

```csharp
// Файл: Form1.cs
public partial class Form1 : Form
{
    // Оголошуємо об'єкт бізнес-логіки
    private DiscountCalculator _calculator;

    public Form1()
    {
        InitializeComponent();
        _calculator = new DiscountCalculator();
    }

    private void btnCalculate_Click(object sender, EventArgs e)
    {
        try
        {
            // 1. Збір та валідація даних інтерфейсу (UI Logic)
            if (!decimal.TryParse(txtTotalAmount.Text, out decimal totalAmount))
            {
                MessageBox.Show("Будь ласка, введіть коректну базову суму.");
                return;
            }

            bool isVip = chkVipCustomer.Checked;

            // 2. Виклик бізнес-логіки! Форма сама нічого не рахує.
            decimal finalPrice = _calculator.CalculateDiscount(totalAmount, isVip);

            // 3. Відображення результату (UI Logic)
            lblFinalPrice.Text = $"Ціна зі знижкою: {finalPrice:C}";
        }
        catch (ArgumentException ex)
        {
            MessageBox.Show(ex.Message, "Помилка валідації");
        }
    }
}
```

### Патерни проєктування (MVP / MVC)

Хоча підхід вище є суттєвим кроком уперед порівняно з написанням всього коду в Form1, при розробці великих та складних проектів йдуть ще далі, розмежовуючи відповідальність через архітектурні патерни, наприклад **Model-View-Presenter (MVP)**.

- У MVP форма (_View_) визначається інтерфейсом та лише повідомляє _Presenter_ (викликає його методи) про те, що користувач щось натиснув або ввів.
- _Presenter_ звертається до _Model_ (нашої бізнес-логіки, сервісів/БД).
- Отримавши результат з Model, _Presenter_ самостійно каже _View_, куди і що написати, використовуючи наданий йому інтерфейс взаємодії. Це робить Windows Forms застосунки максимально протестованими (testable) та чистими.
